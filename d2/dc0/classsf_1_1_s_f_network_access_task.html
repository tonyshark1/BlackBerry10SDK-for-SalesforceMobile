Detailed Description

A class for performing basic HTTP/HTTPS transactions.

This class is a concrete implementation of abstract class SFGenericTask. It's designed to send HTTP request and receive response asynchronously and concurrently. The purpose of this class is to provide a base implementation for any complex network transactions. The class is implemented using Finite-State-Machine(FSM) approach and is capable of running any response processing in a separated thread from QThreadPool.

Usage

For most REST requests with Force.com, developer should use SFRestResourceTask which is a subclass of this. However, this class can be used directly for any non-RESTful transactions:

//Prepare HTTP request
QNetworkRequest request("https://www.webservice.com/path/to/the/service");
QByteArray body;
//Configure your request...
//Prepare the task
SFNetworkAccessTask *task = new SFNetworkAccessTask(getSharedNetworkAccessManager(), request, HTTPMethod::HTTPPost);
task->setRequestBytesArray(body.toUtf8());
//Enable cancel
task->setCancellable(true);
connect(pCancelButton, SIGNAL(clicked()), task, SLOT(cancel()));
//Connect slots, schedule and run
task->startTaskAsync(this, SLOT(onNetworkReplyReady(sf::SFResult*)));

And in your slot:

void SFOAuthCoordinator::onNetworkReplyReady(sf::SFResult* result) {
if (result->hasError()) {
//Handle error
} else {
QNetworkReply *reply = result->payload<QNetworkReply*>();
if (reply) {
//Handle the reply
}
}
}

Subclassing Notes

Typically, for any Force.com REST API, you should use SFRestResourceTask. In addition, the source code of SFRestResourceTask provides a good example of subclassing SFNetworkAccessTask. You can add your custom logic by overriding following functions:

  • prepare() called as soon as the task is started. The function is executed in main thread (or the thread this task belongs to)
  • cleanup() called when the task is finished and ready to deliver the result. The base class implementation does the memory cleanup and emits signals in proper order. This function also attempt to re-try if enabled. So please call base class implementation at the end of your custom logic. This function can be executed in any thread.
  • retry() called when auto re-try is triggered. This function reset the task and cleanup memory allocated during the operation. This function is invoked from cleanup(), so it can be executed in any thread.
  • ensureRequest() called before the request is sent. It is executed only in the task's owning thread. Please see ensureRequest() for possible returned states.
  • preprocessReply(QNetworkReply*) called when each QNetworkReply finished. It is executed only in the task's owning thread. The main responsibility of this function is to handles HTTP redirect or caching. Please see preprocessReply(QNetworkReply*) for possible returned states.
  • processReply(QNetworkReply*) called when the reply is ready to be processed. The function is always executed in a separate thread. Please see processReply(QNetworkReply*) for possible returned states.
